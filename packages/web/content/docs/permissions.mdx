# Permissions & Access Control

Tank permissions are an explicit contract between skill publishers and skill consumers. They are not a runtime sandbox in the current repo.

## Current Permission Types

Permissions are declared in `skills.json`.

| Permission | Meaning |
|------------|---------|
| `network.outbound` | allowed outbound domains |
| `filesystem.read` | readable project-relative globs |
| `filesystem.write` | writable project-relative globs |
| `subprocess` | whether child process execution is allowed |

There is no separate environment-variable permission in the current shared schema.

## Skill Declaration

Example:

```json
{
  "name": "@acme/seo-audit",
  "version": "1.0.0",
  "permissions": {
    "network": {
      "outbound": ["api.example.com"]
    },
    "filesystem": {
      "read": ["./src/**"],
      "write": ["./output/**"]
    },
    "subprocess": false
  }
}
```

## Project Budget

Projects can define a budget in their own `skills.json`. Installed skills must fit within that budget.

```json
{
  "skills": {
    "@acme/seo-audit": "^1.0.0"
  },
  "permissions": {
    "network": {
      "outbound": ["api.example.com", "*.anthropic.com"]
    },
    "filesystem": {
      "read": ["./src/**", "./docs/**"],
      "write": ["./output/**"]
    },
    "subprocess": false
  }
}
```

## Enforcement

Current enforcement points:

1. publish validates the permission shape
2. install and update enforce the project budget
3. scanner compares declared permissions with observed behavior
4. publish checks whether a new version escalates permissions beyond the allowed semver bump

## Escalation Rules

Current publish rules:

- PATCH: no new permissions
- MINOR: no dangerous additions like new network domains or enabling subprocess
- MAJOR: permission changes allowed

## Limits Of The Current Model

Tank does not currently provide OS-level runtime isolation. The protection model today is:

- explicit declaration
- install-time enforcement
- scanner verification
- human review where needed
