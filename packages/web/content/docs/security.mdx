# Security Model

Tank treats AI agent skills as supply-chain artifacts, not harmless documentation. This page describes the protections that are implemented in the current repo.

## Why Tank Exists

Agent skills execute with the agent's authority. A malicious skill can read files, make network requests, or spawn subprocesses through the host agent. That is the threat model.

ClawHavoc matters because it showed what happens when a registry has no meaningful validation, no permission contract, and no security scanning.

## Current Security Layers

Implemented today:

- manifest validation on publish
- SHA-512 integrity verification on install
- permission budgets enforced at install/update time
- permission escalation checks on publish
- 6-stage security scanning service
- stored scan verdicts and findings in PostgreSQL

Planned, not implemented in this repo:

- runtime sandbox enforcement
- signing/provenance beyond the current integrity model

## 6-Stage Scanner

The scanner service lives in `packages/scanner`.

| Stage | Purpose |
|-------|---------|
| 0 | fetch tarball, safe extract, SHA-256 file hashes |
| 1 | structure validation |
| 2 | static analysis and permission cross-check |
| 3 | prompt injection and hidden-content detection |
| 4 | secret scanning |
| 5 | dependency and supply-chain checks |

### Stage 2

Stage 2 combines:

- Python AST analysis and Bandit
- JS/TS regex and AST-style pattern checks
- shell-script pattern checks
- permission cross-checking against declared capabilities

### Stage 3

Stage 3 looks for:

- direct prompt override attempts
- role hijacking patterns
- exfiltration instructions
- hidden content in comments or encoded payloads
- optional corroboration from external scanners or LLM-assisted review when configured

### Stage 4

Stage 4 uses:

- `detect-secrets`
- custom regex patterns for keys, tokens, database URLs, webhooks, and private keys

### Stage 5

Stage 5 checks:

- dependency manifests
- OSV vulnerability data
- typosquat-like names
- suspicious install/runtime dependency behavior

## Verdict Rules

| Condition | Verdict |
|-----------|---------|
| 1+ critical findings | `fail` |
| 4+ high findings | `fail` |
| 1-3 high findings | `flagged` |
| only medium/low findings | `pass_with_notes` |
| no findings | `pass` |

## Install-Time Protections

Tank also protects consumers during install.

- tarballs are verified with SHA-512 integrity
- extraction rejects symlinks, hardlinks, traversal, and absolute paths
- tarball size and file-count limits are enforced independently of publish-time checks

## Permissions

The current permission contract is declared in `skills.json`.

- `network.outbound`
- `filesystem.read`
- `filesystem.write`
- `subprocess`

The CLI enforces a project permission budget before install completes. The scanner then compares observed behavior to declared permissions and surfaces mismatches as findings.
